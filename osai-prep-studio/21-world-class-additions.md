## B. Original R&D extensions

### B1. NVIDIA AI Kill Chain crosswalk
AI-300 M1 explicitly references the **NVIDIA AI Kill Chain** alongside OWASP/ATLAS, yet most prep ignores it. We add it as a **stage spine** that orders the engagement and crosswalks to our taxonomy:

| Kill-chain stage | Studio activity | Labs | OWASP/ATLAS |
|---|---|---|---|
| Recon | enumerate AI surfaces | L08 | `AML.TA0002` |
| Access/Injection | get untrusted input in | L01–L03, L11 | LLM01 |
| Execution | model/agent acts | L05/L06, L13, L16 | LLM05/06 |
| Persistence | survive across sessions | L15 | persistence |
| Exfiltration/Impact | data out / harm | L07, L09/L10, L19 | LLM02/08/10 |

Lessons tag a `kill_chain_stage`; the exam playbook ([00b-exam-blueprint.md](00b-exam-blueprint.md)) is organized by it.

### B2. Purple-team / detection-engineering loop (the repo's unfair advantage)
The studio is built *on a detection engine*. We exploit that: a **build-your-own-detector** track where the learner writes a detector for an attack they just executed and is graded on **precision/recall against a labeled corpus** generated by `../projects/llm-log-triage/src/generate_logs.py`. This closes the loop `attack → detect → report → harden` that no competing prep offers, and directly serves the exam's defensive/remediation expectations.

```yaml
detector_challenge:
  target_attack: "L02 indirect injection"
  corpus: "generated, labeled (adversarial + benign)"
  pass: { precision: ">= 0.9", recall: ">= 0.85", no_regression_on: "benign set" }
  reuse: "tests/ pattern + detector_catalog()"
```

### B3. Attack-with-AI proficiency ladder (graded automation)
Because the exam allows AI, we don't just *permit* `ATTACK_WITH_AI` — we **grade the learner's automation artifact**. A ladder from manual → tool-assisted → fully-automated, scoring the quality of the PyRIT/garak harness they build (coverage, reproducibility, signal hygiene):

| Rung | Skill | Graded artifact |
|---|---|---|
| A1 | Manual exploitation | a working manual payload |
| A2 | Tool-assisted (garak probe / promptfoo case) | a runnable config + triage of results |
| A3 | Orchestrated (PyRIT multi-turn) | an attack orchestrator vs a lab target |
| A4 | Attacker-LLM in the loop | an agent that adapts payloads + a results report |

This is the single most OSAI-specific skill and is largely absent from the market.

### B4. Anti-cheat — per-learner flags & scenario seeding
Flags are not static strings. Each is `flag = HMAC(server_seed, learner_id || lab_id || attempt)`, verified server-side, so a shared answer is worthless. Exam scenarios are **seeded per learner** (target composition, planted-secret values, doc corpus permutation) so two learners get materially different engagements. Leaderboards add collusion detection (improbably-fast solves, identical evidence). Defeats the answer-sharing failure mode that erodes range credibility.

### B5. Local-model hosting & cost engineering
The hardest infra risk (R1) gets a concrete answer:
- **Targets:** small instruct models via **Ollama** (e.g., 7–8B class), CPU-runnable, with **deliberately under-guardrailed system prompts** that mirror `MockTarget` ("blocks the obvious, leaks on the subtle") — exploitable for teaching, not a capable real-world harm engine.
- **No internet egress** from lab containers; context/token caps per lab; `cost_limit_usd` per lab manifest.
- **Routing:** local for routine tutor turns + all lab targets; a hosted frontier model only for report-grading and Socratic depth. The platform **dogfoods its LLM10 detector** for spend caps.
- A published **compute/cost reference** (per-lab token budgets, expected $/learner-hour) so self-hosters can size hardware.

### B6. Learning science & the skill graph
Beyond SRS, apply evidence-based pedagogy: **interleaving** (mix OWASP categories rather than blocking), **desirable difficulties** (hint penalties, no-AI reps), **deliberate practice** on weak skills surfaced by the heatmap, and **cognitive-load-aware** lesson chunking. The **skill DAG** ([05-progress-engine.md](05-progress-engine.md)) with mastery propagation is the backbone: a learner can't "pass" indirect injection while direct injection mastery is low. FSRS parameters are tuned per-skill from cohort retention data.

### B7. Capstone "MegacorpAI" engagement narrative
A single 24-hour story that *stitches* labs into one realistic engagement (not 20 isolated puzzles): a fictional company with an HR RAG assistant, a customer-support agent with MCP tools, a model gateway, and a cloud-hosted inference service — each carrying planted, chainable weaknesses. The intended path chains **recon (L08) → indirect injection (L02) → excessive agency (L16) → MCP pivot (L11/L13) → infra/extraction (L18/L19) → full report**. This is the exam dress rehearsal and the studio's signature artifact ([06-exam-simulator.md](06-exam-simulator.md)).

### B8. Rules-of-engagement & authorization workflow
Real AI red-team work begins with scope and authorization. Learners practice producing an **RoE/scoping document** (targets, exclusions, data-handling, escalation contacts, time windows) before a capstone — graded as a deliverable. This builds the professional habit the exam's "engagement" framing and real jobs demand, and reinforces the authorized-lab-only ethic ([11-safety-legal-ethics.md](11-safety-legal-ethics.md)).

### B9. Agentic / MCP depth beyond the OWASP list
The frontier of the role. We add forward-looking agentic content: **agent identity & delegation** (confused-deputy across agents), **tool-call provenance** (signing/verifying tool definitions to detect shadowing/rug-pull), **inter-agent/A2A trust** (one compromised agent poisoning a swarm), and **planner/critic manipulation** in multi-agent loops. These extend Track 4 and feed labs L11–L15 with sophisticated variants.

### B10. Gold-set psychometrics & difficulty calibration
Treat the gold set as an instrument: calibrate **item difficulty** from learner response data (IRT-lite — discrimination + difficulty per item), retire items that don't discriminate, and ensure each readiness gate's items actually predict lab/exam success. This makes the readiness score *meaningful*, not just additive points.

### B11. Cohort, mentorship & range seasons
A **range season** (CTF-style, fixed window, fresh scenario seeds, fair-play rules) creates urgency and community; cohort dashboards and an opt-in mentor pairing (strong solvers review weaker learners' reports) accelerate the reporting skill specifically. Anti-collusion from B4 keeps seasons honest.

### B12. Telemetry & learning analytics
OpenTelemetry-style spans (à la TruLens) over tutor retrievals, grader verdicts, and lab attempts give content-governance a feedback loop: which lessons precede lab failure, where hints get over-used, which gold-set items are stale. Drives the monthly ritual ([20-instructor-ops-runbook.md](20-instructor-ops-runbook.md)) with data, not guesswork.

### B13. Accessibility & inclusive design
Keyboard-navigable lab runner, screen-reader-friendly transcripts, colorblind-safe heatmaps, captioned diagrams, and a low-bandwidth/no-GPU path (the CPU-only local models of B5) so the range is reachable without a workstation rig.

## C. Forward research agenda (v2+)

Candidate novel labs/features once the spine ships: cross-modal/image prompt injection; agent-swarm goal hijacking; prompt-injection-via-tool-output chains across 3+ hops; embedding-inversion reconstruction; model-fingerprinting via timing/side-channels; automated report-quality coaching; a "defender mode" campaign mirroring each offensive lab; and an open contribution path for community labs (vetted against the lab quality bar, [02-lab-range.md](02-lab-range.md)).

## D. Definition of done & operator handoff

The blueprint revision is complete when: all OffSec claims are confidence-labeled; OWASP mappings use `LLMxx:2025` names; every lab has an attack-path graph, authorized scope, two-signal grading, and ≥1 defense level; every Track-6 lesson maps to business impact; the tutor has source-authority tiers; the gold set includes stale-claim + unsafe-live-target probes; the README matrix includes the readiness/governance docs; and the MVP builds the spine before premium UX. The implementation operator prompt and approval language from the addendum are preserved as the handoff for the (separately greenlit) build phase.
